This rule raises an issue when logging.error() is used within an exception handler (except block) to log exception information.

Why is this an issue?

When handling exceptions in Python, it’s important to capture not just the error message, but also the full traceback. The traceback shows the sequence of function calls that led to the exception, which is essential for understanding what went wrong and where.

The logging module provides two methods that might seem similar:

  • logging.error(): Logs an error message at the ERROR level
  • logging.exception(): Logs an error message at the ERROR level AND automatically includes the full exception traceback

When you’re inside an exception handler (except block), you’re dealing with an exception that just occurred. In this context, logging.exception() is the appropriate choice because:

  • It automatically captures the traceback without requiring exc_info=True
  • It clearly signals to other developers that you’re logging an exception
  • It follows Python’s principle of being explicit about intent

Using logging.error() in an exception handler means you’re either:

  • Missing the traceback entirely (if you don’t pass exc_info=True), making debugging much harder
  • Writing more verbose code (if you do pass exc_info=True) when a simpler, more idiomatic option exists

The logging.exception() method is specifically designed for this use case and is the established Python idiom for logging exceptions.

If omitting the traceback is intentional, explicitly passing exc_info=False (or exc_info=None) makes that intent clear and is therefore not flagged.

What is the potential impact?

Using logging.error() instead of logging.exception() in exception handlers reduces code maintainability and makes debugging more difficult.

When traceback information is missing from logs, developers must spend additional time reproducing issues to understand their root cause. In production environments, where issues may be intermittent or difficult to reproduce, missing traceback information can make it nearly impossible to diagnose problems effectively.

Additionally, using logging.error() with exc_info=True instead of the more idiomatic logging.exception() makes the code less readable and may confuse developers who are unfamiliar with this pattern.

How to fix it

Replace logging.error() with logging.exception() when logging within an exception handler. If you were using exc_info=True, you can remove it since logging.exception() includes traceback information by default.

If you intentionally do not want to log the traceback, explicitly pass exc_info=False to logging.error() to signal that intent.

Code examples

Noncompliant code example

import logging

try:
    raise ValueError("Invalid value")
except ValueError:
    logging.error("Exception occurred", exc_info=True)  # Noncompliant

Compliant solution

import logging

try:
    raise ValueError("Invalid value")
except ValueError:
    logging.exception("Exception occurred")

If logging the traceback is not desired, explicitly opt out with exc_info=False:

import logging

try:
    raise ValueError("Invalid value")
except ValueError:
    logging.error("Exception occurred", exc_info=False)
